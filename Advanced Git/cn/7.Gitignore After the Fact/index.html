    6.事后的Gitignore¶

    当你开始一个新的软件项目时，你可能认为你开始的预制件.gitignore会涵盖所有可能的情况。但更多的时候，你会意识到你已经提交了不应该提交的文件到仓库。虽然看起来你所要做的就是在.gitignore中引用那个文件，但你会发现这并不像你想象的那样能解决问题。

    在这一章中，你将讨论一些常见的情况，在这些情况下你需要回去告诉Git忽略这些类型的错误。你将会看到两种在本地解决这个问题的情形。迫使Git假设一个文件没有变化，以及从Git的内部索引中删除一个文件。

    开始工作¶

    要开始，请解压本章材料中starter目录下的启动器.zip文件中的资源库。或者，如果你已经完成了前一章的所有挑战，可以继续完成。

    .gitignore跨分支¶

    Git 简单而廉价的分支策略很了不起，不是吗？但有些时候，不经深思熟虑就在各分支间翻来覆去，会让你陷入混乱。

    下面是一个常见的场景来说明这个问题。

    在magicSquareJS项目中，你应该已经在main分支上了。如果你不在，运行git checkout main。

    用ls调出顶层目录的列表，你会看到一个名为IGNORE_ME的文件。用cat IGNORE_ME将该文件的内容打印到命令行，你会看到以下内容：

    Please ignore this file. It's unimportant.

    现在假设你在另一个分支有一些工作要做。用以下命令切换到yDoublyEven分支：

    git checkout yDoublyEven

    用以下命令调出一个完整的目录列表：

    ls -la

    你会看到那里有一个.gitignore，但没有看到IGNORE_ME文件。看起来事情到目前为止都在正常运行。

    在编辑器中打开.gitignore文件，你会看到下面的内容：

    IGNORE_ME*

    看起来你都设置好了忽略那个IGNORE_ME文件。因此，如果你创建了一个IGNORE_ME文件，Git应该完全忽略它，对吗？让我们来看看。

    在当前目录下创建一个名为IGNORE_ME的文件，并在该文件中添加以下文字：

    Please don't look in here

    保存你的改动并退出。

    你可以通过执行git status来检查Git是否忽略了该文件：

    On branch yDoublyEven
Your branch is up to date with 'origin/yDoublyEven'.

nothing to commit, working tree clean

    到目前为止，情况良好。看起来一切都在按计划进行。

    现在用下面的命令切换回main：

    git checkout main

    在这一点上，Git应该没有什么可抱怨的，因为它忽略了那个IGNORE_ME文件。但打开那个IGNORE_ME文件，看看里面有什么：

    Please ignore this file. It's unimportant.

    等等，Git不是应该忽略对该文件的修改，保留你在另一个分支上添加的 "请不要看这里 "的原始文本吗？如果Git应该忽略对这个文件的任何修改，为什么还要覆盖你的修改？

    听起来你应该看看main中的.gitignore文件，看看发生了什么。main里有一个.gitignore文件，对吗？

    用ls -la调出一个完整的目录列表，你会发现，事实上，在main分支上没有.gitignore：

    .
├── .git
├── .tools-version
├── IGNORE_ME
├── LICENSE
├── README.md
├── SECRETS
├── css
├── img
├── index.html
└── js

    哦。嗯，这似乎很容易解决。你只需在main上的.gitignore中添加对IGNORE_ME的引用，一切就会自行解决。

    在当前目录下创建一个.gitignore文件，并在其中添加以下内容：

    IGNORE_ME*

    保存你的改动并退出。所以Git现在应该开始忽略对IGNORE_ME的任何修改，对吗？看来你可以把原来的改动放回原处了。

    在编辑器中打开IGNORE_ME，将该文件的内容替换为你最初想要的内容：

    Please don't look in here

    保存你的改动并退出。执行一个快速的git status来检查Git是否真的像你希望的那样忽略了那个文件：

    git status

    你会在控制台看到以下内容，表明Git绝对没有忽略那个文件：

    On branch main
Your branch is ahead of 'origin/main' by 21 commits.
  (use "git push" to publish your local commits)

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
    modified:   IGNORE_ME

Untracked files:
  (use "git add <file>..." to include in what will be committed)
    .gitignore

no changes added to commit (use "git add" and/or "git commit -a")

    等等，什么？你告诉Git忽略那个文件，但Git显然还在追踪它。这到底是怎么回事？难道把东西放在.gitignore里，哎呀，我不知道，不是告诉Git要忽略它吗？

    这是关于Git的最令人沮丧的事情之一；然而，一旦你建立了一个关于发生了什么的心理模型，你就会发现Git正在做它应该做的事情。而且你还会找到一种方法来解决你所遇到的情况。

    Git跟踪是如何工作的¶

    当你在你的仓库里做了一个改动，你就把这个文件的信息添加到了Git的index或cache。这是磁盘上的一个二进制结构，用来追踪你添加到仓库的所有内容。

    当Git需要弄清楚你的工作树和暂存区域之间有什么变化时，它只需将索引的内容与你的工作树进行比较，以确定有什么变化。这就Git如何"知道"哪些是未缓存的，哪些是被修改的。

    但如果你先将一个文件添加到索引中，后又在.gitignore文件中添加一个规则来忽略这个文件，这不会影响Git将索引与工作树进行比较。该文件存在于索引中，也存在于你的工作树中，所以Git不会去检查它是否应该忽略这个文件。当一个文件在你的工作树中，但还没有在你的索引中时，Git才会执行.gitignore过滤。

    这就是上面发生的事情。你在main中把IGNORE_ME文件添加到你的索引中，在你把它添加到.gitignore之前。所以这就是为什么Git继续在IGNORE_ME上操作，即使你在.gitignore中引用了它。

    事实上，有一个很方便的命令，你可以用它来查看Git目前在你的仓库里忽略了什么。不管你信不信，在这本书里你已经用过很多次了。它是简单的git status，但在最后加上了--ignored标志。

    现在就执行这个，看看你的仓库里有哪些被Git忽略了：

    git status --ignored

    我的输出看起来像下面这样：

    On branch main
Your branch is ahead of 'origin/main' by 21 commits.
  (use "git push" to publish your local commits)

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
    modified:   IGNORE_ME

Untracked files:
  (use "git add <file>..." to include in what will be committed)
    .gitignore

Ignored files:
  (use "git add -f <file>..." to include in what will be committed)
    .DS_Store
    js/.DS_Store

no changes added to commit (use "git add" and/or "git commit -a")

    所以Git忽略了.DS_Store文件，正如我的全局.gitignore，但它没有忽略IGNORE_ME。幸运的是，有几种方法可以告诉Git开始忽略你已经添加到索引中的文件。

    手动更新索引¶

    如果你只想让Git忽略这个文件，你可以自己更新索引，告诉Git假设这个文件永远不会再有变化。这是一个既便宜又简单的解决方法。

    执行下面的命令来更新索引，并告诉Git假设当它对这个文件进行比较时，这个文件没有变化：

    git update-index --assume-unchanged IGNORE_ME

    Git不会给你任何关于这个命令的反馈，但再次运行git status --ignored，你就会看到区别：

    On branch main
Your branch is ahead of 'origin/main' by 21 commits.
  (use "git push" to publish your local commits)

Untracked files:
  (use "git add <file>..." to include in what will be committed)
    .gitignore

Ignored files:
  (use "git add -f <file>..." to include in what will be committed)
    .DS_Store
    js/.DS_Store

nothing added to commit but untracked files present (use "git add" to track)

    从技术上讲，Git并没有忽略它，但就所有意图和目的而言，这种方法具有相同的效果。Git不会认为这个文件已经被修改，以达到追踪的目的。

    为了证明这一点，请修改IGNORE_ME并在其末尾添加一些文字，如下所示：

    Please don't look in here. I mean it.

    保存你的改动，退出编辑器，然后再次运行git status --ignored。你会看到Git继续假设该文件没有变化。

    这对于你在仓库中添加占位符或临时文件，但又不想让Git在开发过程中跟踪这些临时文件的变化的情况很有用。或者你只是想让Git暂时忽略这个文件，直到你在以后的重构冲刺中修复它。

    这个解决方法的问题是，它只是一个本地的解决方案。如果你在一个分布式仓库工作，其他人要想忽略这个文件，就必须在自己的克隆中做同样的事情。告诉Git假设一个文件没有变化，只是更新你本地系统上的索引。这意味着这些文件的变化不会被提交--但这也意味着其他人在克隆这个repo时仍然会遇到和你一样的问题。

    事实上，你可能更愿意从索引中完全删除这个文件，而不是让Git对它视而不见。

    从索引中删除文件¶

    当你隐式或显式地要求Git开始追踪一个文件时，Git就会尽职尽责地把这个文件放在你的索引中，并开始观察其变化。如果你很确定你不想让Git再追踪这个文件，你可以自己把这个文件从索引中删除。

    当你从索引中删除一个文件后，Git会按照自然的程序检查工作树与索引的变化，然后查看.gitignore，看是否应该从变化集中排除什么。

    你已经遇到过一个从Git的索引中删除文件的命令。git rm。默认情况下，git rm会将文件从索引和你的工作树中删除。但在这种情况下，你并不想删除工作树中的文件 - 你想保留它。

    要从索引中删除一个文件，但在工作树中保留它，你可以使用--缓存选项，告诉Git只从索引中删除这个文件。

    执行以下命令，指示Git从索引中删除IGNORE_ME。因此，Git将停止跟踪它：

    git rm --cached IGNORE_ME

    Git用一个简单的确认来回应：

    rm 'IGNORE_ME'

    要想知道这是否有效，请再次运行git status --ignored：

    On branch main
Your branch is ahead of 'origin/main' by 21 commits.
  (use "git push" to publish your local commits)

Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
    deleted:    IGNORE_ME

Untracked files:
  (use "git add <file>..." to include in what will be committed)
    .gitignore

Ignored files:
  (use "git add -f <file>..." to include in what will be committed)
    .DS_Store
    IGNORE_ME
    js/.DS_Store

    IGNORE_ME现在显示它既被deleted又被ignored。这怎么可能呢？

    如果你考虑一下Git的视角，这就说得通了：git status将暂存区域，或索引，与HEAD进行比较，看看下一次提交应该是什么。Git看到IGNORE_ME已经不在索引中了。此刻这个文件是否存在于磁盘上，对Git来说并不重要。所以它看到下一次提交将从仓库中删除IGNORE_ME。

    除此之外，在git status中加入--ignored选项，会根据工作树中与.gitignore中的任何过滤器相匹配的文件，建立一个Git现在知道要忽略的列表。

    IGNORE_ME不在你当前的索引中，所以当Git运行它的忽略过滤器时，它看到你在磁盘上有一个名为IGNORE_ME的文件，而这个文件不在你当前的索引中。

    然而，你把这个过滤器放在了你的.gitignore中，所以Git把这个文件添加到了它要忽略的文件列表中。因此，IGNORE_ME既是删除状态（就索引而言），又是忽略状态（就你的.gitignore而言）。

    由于这似乎已经清除了这种情况，你现在可以创建你的下一个提交。但是等等--你是不是忘记了什么？是什么让你一开始就陷入了这种困境？

    对了 - .gitignore还没有被追踪。现在就把这个文件归档：

    git add .gitignore

    并在你再次忘记之前承诺这一改变：

    git commit -m "Added .gitignore and removed unnecessary file"

    请记住，如果别人在你推送了这个提交后克隆了这个repo，他们也会在他们的克隆中丢失这个文件。只要这是你的本意，那就没问题。

    现在，请记住，这并不意味着删除你的文件的所有痕迹--在你的版本库中仍然有一整个提交历史，这些提交都是完整的。如果有人真的想，他们可以回溯历史，找到这个文件的内容。

    要看到这一点，请在该文件上运行git log：

    git log -- IGNORE_ME

    该日志的第二个条目显示如下：

    commit 7ba2a1012e69c83c4642c56ec630cf383cc9c62b (origin/main, origin/HEAD)
Author: Yasmin <yasmin@example.com>
Date:   Mon Jul 3 17:34:22 2017 +0700

    Adding the IGNORE_ME file

    嗯，这似乎不是一个大的问题。那么，如果人们可以看到你添加了一个后来从版本库中删除的文件呢？

    在这种情况下，这并不重要。但是，人们经常提交大量的压缩文件或二进制文件到版本库，直到人们抱怨克隆版本库到他们的本地系统需要多长时间才意识到这一点。

    更关键的是，如果你不小心提交了一个包含API密钥、密码或其他秘密的文件呢？那你就绝对要确保你已经清除了版本库中关于这个文件的任何历史记录。如果有人得到你的API密钥或其他秘密，他们有可能无限制地访问你的一些系统。呜呼。

    重置并不总是解决方案¶

    假设你不想让任何人知道IGNORE_ME的存在。你已经学会了一种改写版本库历史的方法。重置。但这能解决你目前的问题吗？

    为了了解为什么重写不是解决这个问题的好方法，你将在当前版本库上进行交互式重写。这将告诉你在哪些情况下git rebase可能不是重写历史的最佳选择。

    你知道Yasmin在提交哈希7ba2a1012e69c83c4642c56ec630cf383cc9c62b中添加了IGNORE_ME，正如你在上面看到的。所以你要做的就是放弃那个特定的提交，在祖先提交的基础上重新建立其他一切，一切都会好的，对吗？

    但首先。那次提交是否只增加了IGNORE_ME？还是添加了其他文件？你需要在提交之前知道这些。你不能总是相信别人的提交信息。

    看一下这个提交的补丁，看看它到底包含了什么：

    git log -p -1 7ba2a10

    你应该看到以下内容：

    commit 7ba2a1012e69c83c4642c56ec630cf383cc9c62b
Author: Yasmin <yasmin@example.com>
Date:   Mon Jul 3 17:34:22 2017 +0700

    Adding the IGNORE_ME file

diff --git a/IGNORE_ME b/IGNORE_ME
new file mode 100644
index 0000000..28c0f4f
--- /dev/null
+++ b/IGNORE_ME
@@ -0,0 +1 @@
+Please ignore this file. It's unimportant.

    好吧，看来那个提交只是添加了那个文件，正如它在提交中所说。理论上说，你应该可以从repo的历史中删除该提交，一切都应该正常。

    用下面的方法开始一个交互式重建：

    git rebase -i 7ba2a10~

    提交哈希末尾的tilde ~意味着"从这个提交之前的提交开始重定操作"。

    Git会向你展示这次重写的交互式脚本：

    pick 7ba2a10 Adding the IGNORE_ME file
pick 883eb6f Adding methods to allow editing of the magic square
pick e632550 Adding ID to <pre> tag
pick f28af7a Adding ability to validate the inline square
pick c2cf184 Wiring up the square editing and validation

    你需要做的就是放弃第一个提交，对吗？使用你的git-fu技能，键入cw删除第一行的pick命令，并在其位置上输入drop。你的rebase脚本应该是这样的：

    drop 7ba2a10 Adding the IGNORE_ME file
pick 883eb6f Adding methods to allow editing of the magic square
pick e632550 Adding ID to <pre> tag

    按Escape退出插入模式，然后输入:wq和Enter来保存你的工作并继续进行交互式重建。

    ...当然，任何事情都不会像它看起来那么简单。你已经在index.html上遇到了合并冲突：

    Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
error: could not apply f985ed1... Centre align everything

    哦，对了。因为Git实际上是在重放所有其他的提交作为重构的一部分，你会遇到与IGNORE_ME无关的文件的合并冲突。

    你没有考虑到的是7ba2a10 Adding the IGNORE_ME file 的祖先 - 以及此后版本库中发生的事情。

    执行下面的命令可以看到这个提交的全部血腥细节：

    git log --oneline --graph --all

    向下滚动，你会看到提交69670e7 Adding a new secret：

    .
.
.
| * | | e632550 Adding ID to <pre> tag
| * | | 883eb6f Adding methods to allow editing of the magic square
| |/ /
* | | 7ba2a10 Adding the IGNORE_ME file
* | | 32067b8 Adding the structure to the generator
|/ /

    69670e7是7ba2a10的祖先。从那时起，仓库里发生了很多事情。所以当Git回放仓库的历史时，它必须一路回到那个祖先，重放每一个属于那个祖先的提交，并在69670e7的基础上重新建立它--甚至是你已经合并回main的提交。唉。这真的不是你想要的，是吗？

    你可以这些提交并解决它们，但这是一个巨大的工作量，而且有相当大的风险，只是为了摆脱一个文件。

    用下面的命令中止这个正在进行的重构：

    git rebase --abort

    这将使你的分期和工作环境重新回到你之前的位置。

    Note

    对于那些纯粹的人来说，你的工作区和暂存区在回溯过程中实际上没有变化。重置发生在一个临时分离的HEAD空间，你可以把它看作是一个"虚拟"分支，在重置完成之前，它不会被拼接到你的repo上。终止重写只是扔掉了这个临时空间，让你回到没有变化的工作和暂存区域。

    这不是一个可扩展的解决方案--丝毫没有。有一个更好的方法，它被称为git filter-branch。

    使用filter-branch来重写历史¶

    我们先把IGNORE_ME的问题放在一边，你会在本章末尾再来讨论它。现在，你将通过一个类似的文件SECRETS来解决这个问题，在这个文件中，你提交了你不希望公开的文件或其他信息，这就是上述可怕的情况。

    用以下命令打印出SECRETS文件的内容：

    cat SECRETS

    你会看到以下内容：

    DEPLOY_KEY=THIS_IS_REALLY_SECRET
RAYS_HOTTUB_NUMBER=012-555-6789

    你能想象如果这两条信息流传到大街上的混乱局面吗？你需要清理仓库，删除该文件的所有痕迹--同时也要确保仓库已经被重写，以消除该文件曾经过的任何迹象。

    Git中的filter-branch命令可以让你以编程方式重写你的仓库。这与你在交互式重写时的做法类似，但它比在交互式重写时手动调整要灵活和强大得多。

    虽然有很多方法可以运行 filter-branch，但你将采取最直接的方法来删除这个文件。重写你的版本库的暂存区域，或索引。

    首先，快速回顾一下。你还记得如何从索引中删除一个文件吗？没错--git rm --cached将文件从你的暂存区移除，而不是你的工作区。记住这个；你马上就会用到它。

    你需要知道git rm的另一个选项：--ignore-unmatch。

    要知道为什么需要这个选项，在命令行执行以下命令，尝试从索引中删除一个不存在的文件：

    git rm --cached -- NoFileHere

    Git会回应一个fatal的错误：

    fatal: pathspec 'NoFileHere' did not match any files

    因为这是一个致命的错误，所以Git会停下脚步，以所谓的非零退出状态返回；换言之，它出错了。

    为了进一步证明这一点，执行下面的连锁Bash命令，如果第一条命令成功，它将打印出success!。

    git rm --cached -- NoFileHere && echo 'success!'

    Git再次回应单一的致命错误并停止；echo 'success!'从未被执行。很明显，如果git rm在一个文件名上不匹配，它就完了，并立即停止执行。

    为了解决这个问题，--ignore-unmatch将告诉Git报告一个零的退出状态--也就是说，成功完成--即使它没有找到任何文件可以操作。要看这个动作，请执行以下连锁的Bash命令：

    git rm --cached --ignore-unmatch -- NoFileHere && echo 'success!'

    你会看到success!打印到控制台，显示git rm成功退出。

    现在--要把这些知识用于工作。

    执行以下命令，运行git filter-branch，删除违规的文件：

    git filter-branch -f --index-filter 'git rm --cached --ignore-unmatch -- SECRETS' HEAD

    Note

    你会看到一个关于git-filter-branch有问题的警告，来自Git本身的信息建议使用一个叫git-filter-repo的外部工具（可在https://github.com/newren/git-filter-repo）。不过对于我们这个简单的例子来说，git-filter-branch已经足够了，而且命令会在显示警告后几秒钟自动进行。

    把那条长长的命令一点一点地进行下去：

    • 你执行git filter-branch来告诉Git开始重写仓库历史。
    • -f选项的意思是"强制"；这告诉Git忽略之前操作的任何内部缓存的备份。如果你经常使用filter-branch，你会想使用-f选项，以避免每次运行filter-branch时，Git都会提醒你有之前操作的现有备份。
    • 接下来指定--index-filter选项，告诉Git重写索引，而不是直接重写你的工作树（后面有更多介绍）。
    • 然后指定过滤器，或命令，你想在Git重写历史时对每个匹配的提交运行。在这个例子中，你要执行git rm --cached来查找索引中的文件。--ignore-unmatched防止Git在不匹配任何文件时跳出filter-branch。最后，你表示你想删除SECRETS文件。
    • 最后一个选项表示要操作的修订列表。在这里提供一个单一的值，在本例中是HEAD，告诉Git将filter-branch应用于从HEAD到历史上最远的提交祖先的所有修订本。

    Git会吐出多行输出，告诉你它在做什么。以下是我输出的一行；你的输出可能略有不同：

    Rewrite baf24aa0291b3364a0762c36509ab4c433527546 (39/40) (2 seconds passed, remaining 0 predicted)    rm 'SECRETS'

    Git已经从HEAD回溯了每一次提交，执行了指定的git rm命令，然后重新提交了更改。为了证明这一点，寻找SECRETS文件：

    git log -- SECRETS

    你不会得到任何回复，告诉你Git的日志对你所要求的SECRETS文件一无所知。

    现在，你似乎已经删除了这个文件的所有痕迹，但有一个小线索可以告诉别人你已经从仓库中删除了什么。

    添加这个文件的原始提交还在。执行git log --oneline --graph，向下滚动，你会看到添加这个秘密文件的原始提交：

    * | dcbdf0c Adding a new secret

    然后，用以下命令查看提交的补丁：

    git log -p -1 dcbdf0c

    Git会显示元数据，但补丁本身是空的：

    commit dcbdf0c2b3b5cf06eafd5dc6e441c8ab3a1d2ed5
Author: Will <will@example.com>
Date:   Mon Jul 3 14:10:59 2017 +0700

    Adding a new secret

    虽然没人知道其中的秘密是什么，但如果能把这个提交完全删除就好了，因为它是空的。这就像使用filter-branch的另一个选项一样简单：--prune-empty。如果你的作者有先见之明，一开始就告诉你要使用它，那么你就可以把它作为一个选项加入到你的原始命令中。

    但是，Git不是一个复仇的神；你可以再次运行filter-branch来清理事情。执行下面的命令，再次运行你的仓库，删除任何"空"提交：

    git filter-branch --prune-empty -f HEAD

    这只是简单地运行你的仓库，删除任何有空补丁的提交。同样，-f命令强制Git执行filter-branch，不考虑之前filter-branch操作中可能保存的任何备份。

    再用git log --oneline --graph调出你的日志，滚动一下；现在提交已经消失了。

    现在你已经是改写版本库历史的专家了，现在是本章的挑战时间。它将会把事情带入正轨，处理你之前工作的那个可怜的IGNORE_ME文件。

    挑战：从版本库中删除IGNORE_ME¶

    现在你已经学会了如何从版本库中消除一个文件的任何痕迹，你可以回去从版本库中删除所有IGNORE_ME的痕迹。

    你之前从版本库中删除了所有SECRETS的痕迹，但这花了你两个步骤。这里的挑战是如何用一条命令完成同样的工作。

    • 使用git filter-branch。
    • 使用--index-filter来重写索引。
    • 你可以使用类似的git rm命令，但记住，这次你是在不同的文件上进行过滤。
    • 使用--prune-empty来删除任何空提交。
    • 记住，你要把它应用到所有的提交，从HEAD开始，一直到后面。
    • 你需要使用-f来强制执行这个filter-branch操作，因为你已经做了一个filter-branch，而且Git已经为你存储了一个操作的备份。

    Note

    如果Git不同意，请检查你的命令中选项的位置是否正确。

    如果你想检查你的答案，或者需要一点帮助，你可以在本章附带的challenge文件夹中找到这个挑战的答案。

    关键点¶

    • .gitignore通过比较暂存区或索引中的文件与你工作树中的文件来工作。
    • .gitignore不会过滤掉已经存在于索引中的任何文件。
    • git status --ignored显示了Git当前忽略的文件。
    • git upd-index --assume-unchanged <filename>告诉Git总是假设索引中包含的文件不会改变。这是一个快速解决没有被忽略的文件的方法。
    • git rm --cached <filename>从索引中删除一个文件，但在你的工作树中留下原始文件。
    • git rm --cached --ignore-unmatch <filename>将成功，如果git rm没有匹配到索引中的文件，则返回退出代码0。当你把这个命令和filter-branch一起使用时，这很重要。
    • git filter-branch -f --index-filter 'git rm --cached --ignore-unmatch -- <filename>' HEAD将修改版本库中任何匹配的提交，从其内容中删除<filename>。
    • --prune-empty选项将删除版本库中任何在你的filter-branch之后为空的提交。

    接下来去哪？¶

    当你在版本库中提交了一些你不希望出现的东西时，本章所学到的东西通常会对你有帮助。

    相反的情况也很常见。你的版本库中没有什么东西，但你知道那段代码或文件存在于另一个分支，甚至另一个版本库中。

    你已经看到了如何用 filter-branch 有选择地从你的版本库中删除修改。但最终，你会遇到把事情弄糟的情况，你需要一个好的老式的"撤销"按钮来修复事情。同样，Git有多种方法来"撤销"你所做的事情--所有这些你都会在下一章中了解到。
