    8.集中式工作流程¶

    集中式工作流程是使用Git最简单的方法。在这种工作流程中，你直接在主分支上工作，而不是在单独的特性分支中工作。

    在Git中创建分支是非常容易的，所以只有在创建分支会造成不必要的额外开销时才应该跳过创建分支。

    在本章中，你将了解到集中式工作流程适合的场景。还将学习如何处理多个开发者直接向主分支提交时出现的常见情况。

    何时使用集中式工作流¶

    首先提交并推送代码到一个分支的主要原因之一，是为了让其他开发者在推送到主分支之前审查你的代码。如果代码不需要被审查，那么创建和推送一个单独的分支的开销就没有必要。这就是集中式工作流很适合的地方。

    以下是一些可能不需要代码审查的情况。

    1. 独自工作时¶

    如果你是一个项目的唯一开发者，你不需要创建分支的开销，因为没有其他开发者来审查你的代码。

    考虑一下，如果你在合并到main之前将你的特性提交到一个分支，你会运行哪些命令：

    git checkout -b my-new-feature # 1: Create and switch to branch
# Write the code
git add . && git commit -m "Adding my new feature"
git checkout main              # 2: Switch back to main
git merge my-new-feature       # 3: Merge branch into main
git branch -d my-new-feature   # 4: Delete branch
git push main

    与你使用集中式工作流程处理同样的更新相比，你会跳过上面的四个编号的命令，最后只用："我是谁"。你会跳过上面四个编号的命令，最后只有：

    # Write the code
git add . && git commit -m "Adding my new feature"
git push main

    即使在使用集中式工作流程时，仍然有合理的理由来创建分支。例如，如果你有实验性的或不完整的代码，但你还没有准备好提交到主分支，你可以把它提交到特性分支，以后再重新审视它。

    在集中式工作流程中，创建分支是可选的，因为你可以直接将你的提交推送到主分支上。而在下一章将要介绍的特性分支工作流程中，情况就不是这样了。在该工作流程中，创建分支是必须的，因为不允许直接推送到主分支上。

    2.在一个小团队中工作时¶

    如果你是一个小团队的一员，每个团队成员都有专门的知识领域，那么集中式工作流是一个不错的选择。例如，如果一个开发人员使用一种编程语言编写后台代码，而另一个开发人员使用另一种语言编写前端代码，那么让这些团队成员审查他们专业领域之外的代码并不总是有用或实用的。

    在另一种常见的情况下，每个开发人员拥有代码的特定区域。例如，在一个iPhone应用程序中，一个开发人员负责搜索流程，而另一个开发人员负责设置和账户偏好。在这种情况下，团队中的每个成员都完全负责做出他们需要的修改，并确保他们的修改能正常工作。

    3.当优化速度的时候¶

    在将代码推送到中央仓库之前，代码审查是提高代码质量的好方法，但每个代码审查都有一些开销。

    在作者提交他们的修改后，他们需要等待有人来审查，这可能会阻碍他们继续前进。

    此外，关于代码审查的电子邮件和警报是破坏性的。一些团队成员可能会停止他们正在做的事情，快速看一下代码审查请求，看看他们是否可以立即审查它。如果不能，他们就需要花时间来做这件事。在进行重点工作时，如软件开发，上下文切换尤其昂贵。

    任何正在等待审查的代码都会给作者和审查小组的其他成员带来精神负担。

    下面的顺序图说明了一些所需的额外时间和开销。

    第一个红色部分说明了等待你的代码被审查的开销。第二个部分显示了当你中断目前正在进行的工作，回去把原来的代码合并到main时，你必须进行上下文切换。

    代码审查的时间越长，其他人就越有可能引入冲突，而你必须手动解决。

    如果你想优化速度，减少干扰，你的团队可以采取一种策略，在作者将代码推送到主分支之前，不必审查代码。

    请记住，在推送到main之前不审查代码并不意味着团队不能在之后审查代码。这只是意味着main上的代码第一次可能没有那么干净和完美。

    另一方面，即使是经过良好审查的代码也远非完美。在优化速度的时候，为了权宜之计，允许多一点熵可能是有意义的。

    这并不意味着你根本就不能让你的代码被审查。你可以随时创建一个分支，要求对一个新的或复杂的功能进行专门的代码审查。这只是意味着没有一个全面的政策来要求对每个新功能进行代码审查。

    4.当在一个新项目上工作时¶

    当在一个新的项目上工作时，对权宜之计的需求往往更强烈，因为最后期限很紧。在这种情况下，等待代码审查的不便可能会特别多。

    虽然错误在任何情况下都是不可取的，但未发布的项目对它们的容忍度更高，因为它们的影响很低。因此，在推送到main之前，你不需要对每一个提交进行彻底的审查。

    即使你的新项目一开始没有使用集中式工作流，如果你的团队让你在最后期限临近时直接提交并推送到main，也不要感到惊讶

    集中式工作流的最佳实践¶

    这里有一些你可以采用的最佳实践，以使集中式工作流的使用更容易。当在团队中工作时，这些做法尤其重要，因为在团队中，有多个开发人员要提交到main。

    有两件重要的事情需要记住，一是尽早和经常重新发布，二是重新发布比创建合并提交更重要。如果你不小心创建了一个合并提交，只要你没有把它推送到远程仓库，你可以很容易地撤销它。

    尽早和经常地重做¶

    当在团队中使用集中式工作流程时，你经常需要在推送到main之前重新设置，以避免合并提交。

    甚至在你准备推送本地提交的代码到远程仓库之前，你就可以将你的工作重新放到main中的任何新提交的代码上。你可能会拉入一个错误修复，或者拉入你正在构建的功能的代码。

    你越早解决冲突并将你的工作与main中的代码整合，就越容易做到。例如，如果你正在使用一个最近被重新命名的变量或常量，如果你早点把它拉进来，你将会有更少的更新要做。

    记住，你要使用git pull --rebase，而不仅仅是git pull，这样你就可以将本地main分支上的任何提交重新归入origin/main，否则这可能会产生一个合并提交。你很快就会看到这样的例子。

    撤销意外的合并提交¶

    有时，你的本地main分支可能会与远程origin/main分支发生分歧。例如，当你有尚未推送的本地提交，而远程的origin/main有其他人推送的较新提交。

    在这种情况下，执行一个简单的git pull就会产生一个合并提交。合并提交是不可取的，因为它增加了一个额外的不必要的提交，使回顾Git历史变得更加困难。

    如果你不小心创建了一个合并提交，只要你没有把合并提交推送到远程仓库，你就可以轻松地撤销它。

    在本章的项目中，你将通过一个例子来演示这个工作流程，以及如何处理直接在主分支上工作时遇到的一些问题。

    撤销合并提交¶

    由于Alex的main分支与origin/main分支有分歧，运行git pull会导致合并提交。

    这是因为git pull实际上是两个独立命令的组合：git fetch和git merge origin/main。

    如果Alex没有任何本地提交，那么命令中隐含的git merge部分将执行快进合并。这意味着Alex的main分支指针会简单地向前移动到origin/main所指向的位置。然而，由于main已经分流，这就产生了一个合并提交。

    1.终止合并提交¶

    阻止合并提交的最简单的方法是通过让提交信息为空来缩短这一过程。

    从alex/checklists，运行git pull。Vim会打开，显示如下：

    Merge branch 'main' of ../../checklists
# Please enter a commit message to explain why this merge is necessary,
# especially if it merges an updated upstream into a topic branch.
#
# Lines starting with '#' will be ignored, and an empty message aborts
# the commit.

    Note

    如果Vim没有打开，而git pull命令成功了，Successfully rebased and updated refs/heads/main，你可能在你的全局~/.gitconfig文件中将pull.rebase选项设置为true。 在这种情况下，你可以运行git config pull.rebase false来覆盖这个版本库的设置，这样你就可以跟着这一章走了。然后运行git reset --hard 865202c来撤销拉动。现在运行git pull应该会带你到Vim。

    看一下提交信息模板的最后一行。它说：

    Lines starting with '#' will be ignored, and an empty message aborts the commit.

    这意味着你可以输入dd来删除第一行，而保留其余的行，因为它们都以#开头。

    然而，清除完整的提交信息是令人放心的。因为这需要同样多的按键，所以你可以这样做。输入dG删除所有内容，直到最后，然后输入:wq退出。

    现在，你会看到以下内容:

    Auto-merging index.html
error: Empty commit message.
Not committing merge; use 'git commit' to complete the merge.

    Note

    你还会看到一些hint:前缀的文字，关于设置pull.rebase首选项。这是你在稍后的Setting up automatic rebase部分要做的事情。

    正如上面的最后一行所示，你中止了合并的提交，但没有中止合并本身。

    你可以通过运行git status来验证这一点：

    ...
All conflicts fixed but you are still merging.
...

    运行下面的命令来中止合并本身：

    git merge --abort

    恭喜你，避免了合并提交!

    2.硬重置到ORIG_HEAD¶

    再次运行git pull来触发合并。当Vim打开时，输入:wq接受默认信息并提交合并。

    现在运行git log --oneline --graph：

|\

| * ed17ce4 B3: Added "Checklists" heading within body
* | 865202c A3: Added Checklists title within head
|/
* 824f3c7 B2: Added empty head and body tags
* 3a9e970 A2: Added empty html tags
* b7c58f4 B1: Added index.html with <!DOCTYPE html> tag
* a04ae7f A1: Initial Commit: Added LICENSE and README.md

    从视觉上看，你的存储库处于以下状态：

    现在你有一个合并提交，MC，它是origin/main中所有尚未在分支中的内容的组合。在本例中，MC包含Beth的B3和B4提交的代码。

    只要你没有把合并提交推送到main，你就可以撤消它。不过，首先要确定Alex的main分支在合并前的提交哈希值，然后用这个提交哈希值运行git reset --hard。

    确定提交哈希值的方法之一是查看提交日志。你可以看到865202c是A3提交的哈希值，也就是合并前main的位置，所以你可以运行git reset --hard 865202c。

    还有一个更简单的方法来识别合并前的提交哈希值。当Git提交一个合并操作时，它会将合并前的原始提交哈希值保存到ORIG_HEAD中。

    如果你很好奇，你可以运行以下任一命令，看看ORIG_HEAD的提交哈希值是多少：

    git rev-parse ORIG_HEAD

    或：

    cat .git/ORIG_HEAD

    这显示了以下情况：

    865202c4bc2a12cc2fbb94f5980b00457d270113

    运行以下命令来执行复位：

    git reset --hard ORIG_HEAD

    你应该看到以下确认信息：

    HEAD is now at 865202c A3: Added Checklists title within head

    你又回到了你开始的地方，这正是你想要的!

    3.重置合并后的提交¶

    另一种策略是将包含合并提交的main分支重新归入origin/main。这样就把A3和合并提交放在B4之上。由于origin/main已经有了B3和B4，也就是合并提交的内容，这样就完全删除了合并提交。

    使用git pull -rebase¶

    你以前学过git pull是两个独立命令的组合：git fetch和git merge origin/main。

    在git pull中添加--rebase选项，实质上是将第二个git merge origin/main命令改为git rebase origin/main。

    运行git pull --rebase。你会看到以下内容：

    Successfully rebased and updated refs/heads/main.

    然后运行git log --oneline --graph来看看提交历史：

    * c1f9be5 (HEAD -> main) A3: Added Checklists title within head
* 4da1174 (origin/main) B4: Added "Welcome to Checklists!" within body
* ed17ce4 B3: Added "Checklists" heading within body
...

    你可以看到，你现在已经将本地A3的提交重定向到origin/main。

    在下一个练习中，最后一次重置你的main分支：

    git reset --hard 865202c

    设置自动重定位¶

    你可能偶尔会在运行git pull之前忘记你在main上有本地提交，从而导致合并提交的发生。当然，这不再是一个可怕的问题，因为你现在知道如何中止和撤消合并提交。

    但如果Git能自动为你解决这个问题，那不是很好吗？它可以做到! 通过在你的Git配置中设置pull.rebase选项为true，你就可以做到这一点。

    这就是前面提示信息中的中间选项：

    hint: Pulling without specifying how to reconcile divergent branches is
hint: discouraged. You can squelch this message by running one of the following
hint: commands sometime before your next pull:
hint:
hint:   git config pull.rebase false  # merge (the default strategy)
hint:   git config pull.rebase true   # rebase
hint:   git config pull.ff only       # fast-forward only
hint:

    你也可以把pull.rebase设置为false，这是默认的，但通过设置，你让Git知道"是的，Git，这确实是我想做的，所以你不用提醒我！"

    另外，你也可以把pull.ff设置为only，这意味着你将明确地自己获取和重设，也就是说，你将运行git pull作为快捷方式的两个命令。如果设置了这个选项，并在不可能进行快进合并的情况下尝试git pull，你会得到一个错误：fatal: Not possible to fast-forward, aborting.。如果你想试试这个选项，你可以，然后运行git config --unset pull.ff来撤销它。

    回到你真正要设置的选项，运行下面的命令来设置Git在运行git pull时总是重新设置：

    git config pull.rebase true

    现在运行git pull。

    Successfully rebased and updated refs/heads/main.

    你会看到它自动在origin/main的基础上重新提交，而不是创建一个合并提交。你可以运行git log --oneline --graph来验证这一点。

    现在，终于到了你一直在努力争取的时刻了! 运行git push来推送Alex的新提交到远程的main分支。

    git push

    你会看到确认它推送到了checklists.git远程：

    ...
To ../../checklists.git
   4da1174..c9266b9  main -> main

    瞧！你现在可以git pull而不必记住添加--rebase选项了。你现在可以git pull而不必记住添加--rebase选项。

    由于git config将选项写入本地.git/config文件，你团队中的每个开发者都必须为自己配置这个选项。

    而且git config只会为你运行的版本库配置这个选项。如果你想为你电脑上的所有仓库配置这个选项，你可以用--global标志运行该命令，即git config --global pull.rebase true。这将在你的全局~/.gitconfig文件中存储该偏好，这样你就不必为每个仓库单独设置。

    关键点¶

    • 当单独工作或小团队工作时，当优化速度或在一个新的、未发布的项目上工作时，集中式工作流程很适合。
    • 你仍然可以为正在进行的代码或临时性的代码审查创建分支。
    • 频繁地重新发布，以纳入上游的变化，并更快地解决冲突。
    • 最好用git pull --rebase代替git pull以避免创建合并提交。
    • 在你的Git配置中把pull.rebase选项设置为true，以便在拉取时自动重新设置。
    • 只要你没有推送到远程仓库，就有多种方法来撤销意外的合并提交。

    现在你已经很好地掌握了集中式工作流的使用方法，Git之旅的下一步就是分支式工作流了 :] 。进入下一章，开始吧
