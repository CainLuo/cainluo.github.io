    5.重放改写历史¶

    正如你在前一章中所看到的，重新发布为你提供了合并的一个很好的选择。但重放也让你有能力重新组织你的版本库的历史。你可以重新排序，重写提交信息，如果你愿意，甚至可以把多个提交压缩成一个整齐的提交。

    就像你在推送本地分支到远程版本库之前会整理你的代码一样，重新发布可以让你在推送到远程之前清理你的提交历史。这让你可以自由地在本地提交你认为合适的内容，然后重新排列和组合你的提交内容，形成一些有语义的提交内容。这些提交对于那些在未来某个时间点需要梳理版本库历史的人（甚至是你自己！）来说，会有更大的价值。

    Note

    再次警告。这种方式最好用于你没有与其他人共享的分支。如果你必须对一个与他人共享的分支进行重定向，那么你必须与每个克隆了该版本库的人达成安排，确保他们都能得到你的分支的重定向副本。否则，到最后你将会面临一个非常复杂的版本库清理工作。

    开始时，只要你完成了前面的挑战，就可以继续使用前一章中的magicSquareJS仓库。否则你可以使用本章素材文件夹中starter目录下的项目。

    重新排序提交¶

    你先看一下Will的wValidator分支。执行以下命令，看看当前的历史记录是什么样子的：

    git log --oneline --graph wValidator

    你会在历史图表的顶部看到以下内容：

    * dc24e14 (HEAD -> wValidator) Updated team acronym to teamWYXZC
* 72c4e8f Added Chris as a new maintainer to README.md
* 4c5274c Refactoring the main check function
* f0f212f Removing TODO
* d642b89 check04: Checking diagonal sums
* 791744c util06: Adding a function to check diagonals
* 556b640 check03: Checking row and column sums
* 75a6d1f util05: Fixing comment indentation
* 72ec86a util04: Adding a function to check column sums
* 34a656c util03: Adding function to check row sums
* 0efbaf2 check02: Checking the array contains the correct values
* 136dc26 (origin/zValidator) Refactoring the range checking function
* 665575c util02: Adding function to check the range of values
* 0fc1a91 check01: checking that the 2D array is square
* 5ec1ccf util01: Adding the checkSqaure function
* 69670e7 Adding a new secret

    这不是可怕的，但这绝对需要进行一些清理。你的任务是将README.md的这两个琐碎的更新合并为一个提交。然后将五个util*提交和三个check*提交重新排序，最后将这些相关的提交合并为两个独立的、整洁的提交。

    交互式重定位¶

    首先。将两个顶级提交合并为一个，放在当前分支内。你已经熟悉了在其他分支上重设分支的方法，但在本章中，你将在同一分支中的其他提交上重设提交。

    事实上，由于一个分支只是一个提交的标签，在其他分支的基础上重放分支，实际上就是在彼此的基础上重放提交。

    但是，由于你想沿途操纵仓库的历史，你不希望Git只是在其他提交之上重放提交。取而代之的是，你将使用interactive rebase 来完成这项工作。

    首先，制定你的游戏计划。你想把前两个提交合并成一个提交，给这个新的提交一个明确的信息，然后把这个被压扁的提交重新放置在原来提交的祖先之上。所以你的计划看起来有点像下面这样：

    • 压下dc24e14和72c4e8f。
    • 为这个被压制的提交创建一个新的提交信息。
    • 在4c5274c的基础上重新建立新的提交。

    要开始一个交互式的重写，你需要使用-i (--interactive)标志。就像以前一样，你需要告诉Git你想在哪上面重构；在这个例子中，是4c5274c。

    你应该已经在wValidator分支上了。执行下面的命令，开始你的第一次Git交互式重写：

    git rebase -i 4c5274c

    Git会打开你系统上的默认编辑器，可能是Vim，并显示如下：

    pick 72c4e8f Added Chris as a new maintainer to README.md
pick dc24e14 Updated team acronym to teamWYXZC

# Rebase 4c5274c..dc24e14 onto 4c5274c (2 commands)
#
# Commands:
# p, pick <commit> = use commit
# r, reword <commit> = use commit, but edit the commit message
# e, edit <commit> = use commit, but stop for amending
# s, squash <commit> = use commit, but meld into previous commit
# f, fixup [-C | -c] <commit> = like "squash" but keep only the previous
#                    commit's log message, unless -C is used, in which case
#                    keep only this commit's message; -c is same as -C but
#                    opens the editor
# x, exec <command> = run command (the rest of the line) using shell
# b, break = stop here (continue rebase later with 'git rebase --continue')
# d, drop <commit> = remove commit
# l, label <label> = label current HEAD with a name
# t, reset <label> = reset HEAD to a label
# m, merge [-C <commit> | -c <commit>] <label> [# <oneline>]
# .       create a merge commit using the original merge commit's
# .       message (or the oneline, if no original merge commit was
# .       specified); use -c <commit> to reword the commit message
#
# These lines can be re-ordered; they are executed from top to bottom.
#

#

    嗯，这是不一样的! 你以前也见过Vim在创建提交信息方面的行动，但这是个新东西。这到底是怎么回事？

    在交互式回溯中压制¶

    在这里，Git把所有经过你的rebase点的提交，72c4e8f和dc24e14，放在了文件的顶部，下面还有一些相当有用的注释。

    在这一步，你所做的是有效地创建一个命令脚本，让Git在重写时遵循。Git会从这个文件的顶部开始向下工作，依次应用它发现的每个动作。

    要执行提交的压扁，你只需将squash命令放在你想压扁到前一个提交的那一行。在本例中，你想把最后一次提交的dc24e14压入72c4e8f。

    Note

    Git交互式rebase以升序提交的方式显示所有提交。这与你习惯于用git log看到的顺序不同，所以要注意你压制的方向是否正确!

    既然你回到了Vim中，你就必须使用Vim命令来编辑文件。用j把你的光标移到dc24e14行的开头，然后按c键，接着按w键--这是"换词"命令，它基本上是删除你光标所在的词，让你进入插入模式。

    所以在这里输入squash。你的文件的前几行现在应该是这样的：

    pick 72c4e8f Added new maintainer to README.md
squash dc24e14 Updated team acronym

    这就是你需要做的，所以写下你的修改，然后用熟悉的Escape + :wq + Enter组合退出。

    然后Git会把你直接扔回另一个Vim编辑器，这个编辑器更熟悉一些：

    # This is a combination of 2 commits.
# This is the 1st commit message:

Added Chris as a new maintainer to README.md

# This is the commit message #2:

Updated team acronym to teamWYXZC

# Please enter the commit message for your changes. Lines starting
# with '#' will be ignored, and an empty message aborts the commit.
#
# Date:      Sat Aug 21 23:20:17 2021 -0500
#
# interactive rebase in progress; onto 4c5274c
# Last commands done (2 commands done):
#    pick 72c4e8f Added Chris as a new maintainer to README.md
#    squash dc24e14 Updated team acronym to teamWYXZC
# No commands remaining.
# You are currently rebasing branch 'wValidator' on '4c5274c'.
#
# Changes to be committed:
#       modified:   README.md
#

    好吧，这是一个提交信息编辑器，你以前也见过。在这里，Git很有帮助地分享了受这次重设操作影响的所有提交的信息。你可以选择保留或编辑这些提交信息中的任何一条，也可以选择创建自己的。

    创建压扁的提交信息¶

    在这种情况下，你就创建自己的吧。按如下步骤清除这个文件。

    1. 输入gg以确保你在文件的第一行。
    2. 输入dG（大写的"G"），从文件中删除所有下面的行。

    现在你有了一个漂亮的、干净的提交信息的文件。按i进入insert mode，然后添加以下信息：

    Added Chris to README and updated team acronym

    然后用Escape + :wq + Enter保存你的改动，继续进行重定位操作。

    Git会继续进行，在最后发出一点成功信息的输出：

    Successfully rebased and updated refs/heads/wValidator.

    执行以下命令，看看当前分支的版本库历史是什么样子：

    git log --oneline --graph

    看看最上面的两行，你会看到以下内容（当然，你的哈希值会有所不同）：

    * 6c07391 (HEAD -> wValidator) Added Chris to README and updated team acronym
* 4c5274c Refactoring the main check function

    Git已经按照您的要求做了；它从这两个旧的提交创建了一个新的提交，并在祖先的基础上重新建立了这个新的提交。要看这两个提交合并成一个的效果，请看Git用下面的命令为被压缩的提交创建的补丁：

    git log -p -1

    看一下该输出的底部，你会看到以下内容：

    -This project is maintained by teamWYXZ:
+This project is maintained by teamWYXZC:
 - Will
 - Yasmin
 - Xanthe
 - Zack
+- Chris

    将这两个提交合并为一个提交，并在祖先提交的基础上重新排序，这就是合并的效果。

    重新排序的提交¶

    开发的异步性和混乱性意味着有时你需要重新安排提交的顺序，以便以后更容易压制一组提交的内容。交互式重构可以让你在一个分支中重新安排提交的顺序。你可以根据需要经常这样做，以保持你的版本库历史的干净。

    执行下面的命令可以看到你的分支的最新提交：

    git log --oneline --graph

    看看最近十几个提交的顺序吧：

    重写提交信息¶

    如果你看一下util01的提交信息，你会发现它被误写成了"Sqaure"而不是"Square"。作为一个词呆子，我不能让它保持原样。但我可以快速地使用交互式rebase来改变这个提交信息。

    Note

    在我的repo中，提交的哈希值为92b5982，而在你的系统中，它可能会有所不同。只需将下面的哈希值替换成你想在其上重新发布的提交的哈希值--也就是说，就在你想改变的提交之前，事情就能顺利进行。

    执行下面的命令，启动另一个交互式重写，指出你想在上面重写的提交。在这个例子中，你想在check01: checking that the 2D array is square的提交之上进行重写：

    git rebase -i 6adde96

    当Vim出现时，你会看到你想改变的提交在列表的顶部：

    pick 92b5982 util01: Adding the checkSqaure function

    确保你的光标在那一行，然后输入cw来剪切pick这个词，并切换到Vim的插入模式。在pick的位置上输入reword，这样Git就会在运行rebase脚本时提示你重写这个提交。

    完成后，脚本的第一行应该如下：

    reword 92b5982 util01: Adding the checkSqaure function

    Note

    在这一步中，你不需要修改提交信息；而是等待Git在运行rebase脚本时提示你这样做。

    用Escape + :wq + Enter保存你的工作，你将立即回到Vim。这一次，Git要求你实际修改提交信息。

    按i进入插入模式，把光标移到那个严重的拼写错误处，把checkSqaure改为checkSquare，然后用Escape + :wq + Enter保存你的工作。

    Git完成了重定位，并把你扔回了命令行。

    你可以通过执行git log --oneline --graph并向下滚动找到你的新的、重新建立的提交，来了解Git已经为你改变了提交信息。另外请注意，之后的所有提交也都有了新的提交哈希值。

    可以肯定的是，固定的拼写是一件小事，但它是一件好的事情。

    压制多个提交¶

    现在你已经把你的实用函数都安排好了，你可以继续把这些提交压缩成一个。

    同样地，你要启动一个交互式的rebase会话，并输入你想在其上重做的提交的哈希值。你想在Adding a new secret的提交之上进行重写，它仍然是69670e7。请记住。当你在一个提交的顶部重载时，该提交并没有改变，所以它的哈希值仍然和以前一样。只是后面的提交会得到新的哈希值，因为每个提交都被重新建立了。

    要开始你的压制冒险，请执行下面的命令，以启动另一个交互式重写：

    git rebase -i 69670e7

    一旦你回到Vim中，找到实用函数的连续提交列表。

    要压制一个提交列表，找到你想压制的序列中的第一个提交，并保持该提交不变。然后，在接下来的每一行，把pick改为squash。当Git执行这个rebase脚本时，每次遇到squash，它都会将该提交与前一行的提交合并。

    这就是为什么你需要保留第一行不动。否则，Git会将那第一次提交压入前一次提交，这不是你想要的。你想把这组与实用功能相关的修改作为一个整洁的单元来压制，而不是把它们压制在前面的某个随机提交中。

    Note

    你可以用一点Vim的方法来加快事情的进展。

    • 在你想压制的第一个提交（util02）上输入cw，把pick改为squash。
    • 然后按Escape回到命令模式。
    • 把光标移到你想压制的下一个提交的开头，然后输入.--一个句号。这告诉Git"再做一次同样的事情，只是在这一行"。

    继续以这种方式处理所有的实用函数提交。完成后，你的重写脚本应该是下面的样子：

squash d761cfe util03: Adding function to check row sums
squash 98c44f8 util04: Adding a function to check column sums
squash 01688ee util05: Fixing comment indentation

    挑战1：更多的压制¶

    你想把所有的check0x提交压缩成一个整洁的提交。你可以按照上面的模式，先在一个rebase中重新整理提交，然后在另一个rebase中进行压扁处理。

    但你也可以在一次rebase中完成这些工作：

    1. 弄清楚你的基础祖先是什么，以便进行重新定位。
    2. 开始一个交互式的重新定位操作。
    3. 重新排列check0x的提交。
    4. 将pick重写脚本命令改为squash，从check02提交开始，一直到Refactoring the main check function提交。
    5. 在Vim中保存你的工作并退出。
    6. 在Vim中为squash操作创建一个提交信息。
    7. 看看你的Git日志，看看你所做的修改。

    挑战2：把你的修改重新放到主干上¶

    现在你已经把你的工作压缩到只有几个提交，是时候让wValidator回到main分支中了。你的第一反应可能是把wValidator合并到main。然而，此时你已经是一个重构大师了，所以你会把这些提交重构到main之上：

    1. 执行git rebase，将main作为rebase目标。
    2. 糟糕--有冲突。打开README.md，解决冲突以保留你的修改，并将修改移到## Contact部分。
    3. 保存你的工作。
    4. 用git add README.md把这些改动归档。
    5. 用git rebase --continue继续进行重构。
    6. 检查日志，看看main指向哪里，wValidator指向哪里。
    7. 检查main分支。
    8. 对wValidator执行git merge。这个合并有什么特别之处，让你避免合并提交？
    9. 删除wValidator分支。

    如果你遇到困难或需要任何帮助，你可以在本章的challenge文件夹中找到这些挑战的解决方案。

    关键点¶

    • git rebase -i <hash>启动一个交互式的重写操作。
    • Git中的交互式重定向让你创建一个"脚本"来告诉Git如何执行重定向操作
    • pick命令意味着在rebase中保留一个提交。
    • squash命令意味着将此提交与前一个提交合并在一起。
    • reword命令可以让你对某个特定的提交信息进行改写。
    • 你可以在rebase脚本中移动行数来重新排列提交的顺序。
    • Rebasing为rebase脚本中的每个原始提交创建新的提交。
    • Squashing可以将多个提交合并为一个提交，并添加新的提交信息。这有助于保持你的提交历史的干净。

    接下来去哪？¶

    交互式重定向是Git最强大的功能之一，因为它迫使你从逻辑上思考你所做的改动，以及这些改动在其他人看来是怎样的。就像你会喜欢克隆一个repo并看到一个漂亮的、说明性的项目历史一样，在你之后的开发者也会喜欢。

    在下一章，你将继续使用rebase来解决一个非常普遍的问题：当你已经提交的文件被Git忽略时，你该怎么办？如果你的开发生涯中还没有遇到这种情况，相信我，你会的。相信我，你会遇到这种情况的。而且，如果不知道如何使用rebase，要解决这个问题是很困难的。
