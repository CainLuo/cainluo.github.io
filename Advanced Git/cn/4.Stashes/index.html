    3.藏匿¶

    在使用Git一段时间后，您可能会开始感到有点拘束。当然，分段、提交、分支和合并这些功能都很好，这些特性毫无疑问地支持现代迭代开发的非线性过程。但是提交仍然是Git的原子级别;下面什么都没有。

    你可能会觉得Git在强迫你进入一个工作流程，在这个流程中，你不能做任何提交以外的事情，你必须限制自己在每次推送工作到仓库的时候都要建立一个有记录的提交。说到性能焦虑症!

    但Git认识到，你不可能总是一直在为提交而工作。开发是混乱和不可预测的；在找到解决一个错误的正确方法之前，你可能会进入几个平行的兔子洞，追寻不同的可能性。或者，很常见的是，你可能正在为你的应用程序建立下一个伟大的功能，突然，你需要切换到一个错误修复分支，并在一个小时内得到一个热修复。这是肯定的，Codus interruptus。

    那么，在上述情况下，你该如何处理那些尚未完成的、无法编译的代码，这些代码还没有准备好提交，但你不想失去它们？

    好吧，如果你是一个像我一样的偏执狂，你可能已经养成了一个过去的习惯，那就是复制你想以后保留的代码目录，就像这样：

    book_repo
├──git_book_before_edits
├──git_book_old
├──git_book_old_epub_test
├──git_book_older
└──git_book_version_with_jokes_removed

    如果你把同样的思维过程带入到Git的工作中，你会经常想做类似的事情，在你正在工作的分支中创建"临时提交"，这样你就不会在Git仓库的范围内丢失你的工作：

    | * b10bc04 It’s the finalllllllll countdooooooooown do-do-do-doooooooooooo
| * 5f20836 Minor tweak
| * 36ff9ca fix lldb thing from errata forums
| * 0ce8469 Moved Mach-O section to DO_NOT_ADD_THIS… will keep it around for...
| * 79c465d removed TODO
| * cebc416 ok, time to wrap this crap up, Derek
| * 42d40dd wow that sucked
| * e7e8228 omg apple changing everything internally
| * 6d162de Partial Revert "wooooooooooooot done w/ code signing"
| * 9676642 wooooooooooooot done w/ code signing
| * 986e288 ok, hopefully no more bugs on that script….
| * da5e21f Fix xattr bug
| * 52a2bc7 dsresign really looking good now
| * 6382f56 more tweaks
| * 66f0041 dsresign close to complete?
| * ba0bcf9 Added dsresign
| * 5ac42d4 meh
| * a5f08df Pre-delete a big section
| * 1d86a71 wow…. much words. Time to delete some content
.
.
.

    (在此向伟大的调试大师Derek Selander表示歉意，他在https://github.com/DerekSelander。爱你，Derek）。

    你可以看到为了"保存"你的工作而进行临时提交是如何迅速失控的。很明显，必须有一个更好的方法来快速藏起正在进行的工作，并在稍后取回它，当你完成了胶带热修复的编码，准备回到重构你留在特性分支上的混乱代码时。

    事实上，Git开箱即提供了这个功能，它被称为git stash。

    介绍git隐藏¶

    让我们回到威尔、赞斯、亚斯敏和扎克组成的紧密团队。今天的工作进展顺利，您已经有时间处理非常重要的README文件。

    Note

    你需要完成上一章中的挑战才能沿着。 如果还没有，可以使用本章材质文件夹中的初学者项目。

    在magicSquareJS存储库中，打开README.md并在文件底部添加一些文本：

    ## Contributing

To contribute to this project, simply

    但在你完成这个想法之前，赞斯给你发了信号。“你能快速浏览一下xUtils分支吗？”她问。而且，作为一个反应迅速、敏捷的团队成员，你同意看一看。

    但你不想失去你所做的一切。当然，这不是很多，但它确实捕捉到了你当时的思想状态，你真的不想重做这项工作。因此，将您的工作保存到README.md并退出文本编辑器。

    回到命令提示符，切换到xUtils分支：

    git checkout xUtils

    Git会检测到你的工作树中有没有任何地方捕捉到的更改，至少在Git看来是这样：

    error: Your local changes to the following files would be overwritten by checkout:

    隐藏的工作原理¶

    Git的方式与您过去创建代码项目的所有“备份”目录的方式非常相似，它允许您在工作树中获取当前的更改集，并将其隐藏在本地存储库中，以便稍后检索。

    列出储藏库¶

    Git允许你创建一个以上的储藏库，并把它们放在一个堆栈中，这样你就可以很容易地找到你要应用的那个。但是，首先，你将创建另一个储藏库来说明这一点。

    Saved working directory and index state WIP on main: f368651 Merge branch 'xReadmeUpdates'

    Note

    既然你是在使用堆栈，似乎Git应该让你使用常见的堆栈操作，事实上，它确实如此。git stash命令是git stash push的方便别名，用于在堆栈中快速创建一个非命名的堆栈。你还可以使用常见的堆栈操作符，如pop、show和list，你会在下面的章节中看到。

    要查看堆栈，可以使用git stash的list选项：

    git stash list

    Git会向你展示它所知道的所有藏品：

    stash@{0}: WIP on main: f368651 Merge branch 'xReadmeUpdates'
stash@{1}: WIP on main: f368651 Merge branch 'xReadmeUpdates'

    由于这是一个堆栈，索引为0的藏书，由stash@{0}标签表示，是最近的，而索引为1的藏书，由于它在堆栈中的位置较远，所以比较老。

    现在，默认的存储空间信息并不具有描述性；如果你一次只处理一个存储空间，这可能就足够了。但是如果它们都有相同的储藏信息，你就不会总是记得最近的储藏是什么。但是就像你可以在创建提交时提供一个消息一样，你也可以在创建储藏库时指定一个消息。

    给储藏库添加消息¶

    在你的项目工作树上做另一个改动，用下面的链式命令创建一个临时文件：

    mkdir temp && touch temp/.keep && git add .

    Note

    bash或Zsh中的&&操作符可以让你按顺序执行命令，但前提是前面的命令已经成功（也就是退出代码为0）。所以，在这个例子中，你试图创建一个名为temp的目录；如果成功了，你就在temp目录下创建一个.keep文件，这样Git就能识别这个目录。如果文件创建成功，你就调用git add .把这个文件放到舞台上，这样Git就能跟踪它。

    现在你的工作树上有了一个被追踪的新增文件，你可以创建一个带有适当信息的储藏库，以帮助你追踪什么是什么。执行下面的命令：

    git stash push -m "Created temp directory"

    在这种情况下，你使用了push操作符，因为git stash本身不允许你提供任何参数。现在，用git stash list再次拉出堆栈，看看现在的堆栈是什么样子：

    stash@{0}: On main: Created temp directory
stash@{1}: WIP on main: f368651 Merge branch 'xReadmeUpdates'
stash@{2}: WIP on main: f368651 Merge branch 'xReadmeUpdates'

    这更具有指导意义，但你的记忆力很差。如果你想不起来你在某个特定的堆栈中到底做了什么，你可以用下面的命令偷看该堆栈条目的内容：

    git stash show stash@{0}

    Git会简要地告诉你这个隐藏的快照有什么变化：

     temp/.keep | 0
 1 file changed, 0 insertions(+), 0 deletions(-)

    事实上，你可以在git stash show上使用git log中的大部分选项，因为Git只是用它自己的日志来显示快照中的变化。你可以像使用git log一样，使用p选项查看你第一个快照的差异，或补丁：

    git stash show -p stash@{2}

    弹出储藏室¶

    现在，你想回到堆栈底部的储藏室所代表的状态：stash@{2}。如果你还记得你在数据结构和算法课程中的堆栈操作，你就会知道你通常会从这个列表的顶部pop一些东西。事实上，你可以用Git来做这件事，用git stash pop来从堆栈中移除最上面的藏书，并将该补丁应用到你的工作环境中。

    然而，在这种情况下，你想要的储藏库并不在顶部，而是在堆栈的底部。而且你已经决定，你也不想再保留其他两个库了。

    Git允许你从堆栈中挑出一个特定的储藏库，并通过git stash apply stash@{number}将其应用到你的工作树中，其中数字是零索引。由于你想应用堆栈底部的藏书，也就是第三项，你应该使用stash@{2}作为git stash apply命令的参数。

    运行下面的命令来应用堆栈中的第三个项目：

    git stash apply stash@{2}

    合并与缓存的冲突¶

    由于从Git的角度来看，应用储藏库和合并提交有很多相似之处，你可能会想，在合并储藏库时是否也会产生冲突。好吧，也许你没有想过这个问题，但我敢打赌，你现在想知道了！答案是肯定的。

    答案是肯定的--在处理储藏库的时候，你肯定会遇到合并冲突。现在你将设置一个场景，告诉你如何解决与储藏库的合并冲突。

    回顾一下，当Git检测到一个文件内的共同子序列有变化时，就会发生合并冲突。在你的案例中，说明这个问题的最好方法是在一个分支和一个储藏室之间改变README文件中的同一行。

    挑战：清理剩余的藏品¶

    还有什么要做的吗？执行git stash list，你会发现那里还有一个藏书。但你已经弹出了那个储藏室，不是吗？当然是的。

    你的挑战是双重的，可能需要潜心研究一下Git文档。

    1. 解释为什么在堆栈中仍然有一个缓存，即使你执行了git stash pop。
    2. 在不使用git stash pop或git stash clear的情况下，从堆栈中移除剩余的条目。还有一种方法可以用git stash从堆栈中删除条目--是什么？

    如果你被卡住了，或者想检查你的解决方案，你可以在本章的challenge文件夹下随时找到这个挑战的答案。

    关键点¶

    • 缓存让你可以存储你正在进行的工作，并在以后检索它，而不用提交到分支。
    • git stash将你的本地修改的当前状态拍成快照，并将其保存为一个缓存。
    • Git以堆栈的形式保存缓存；所有连续的缓存都被推到堆栈的顶部。
    • Git缓存的工作方式与提交很相似，它被捕获在.git子目录内。
    • git stash是git stash push的方便别名。
    • git stash list显示你所做的堆栈。
    • `git stash show 揭示了某一特定储藏库中的修改的简要摘要。
    • git stash show -p <stashname> 显示了特定储藏库中的修改的补丁或差异。
    • git stash apply <stashname> 将某个储藏库的补丁合并到你的工作环境。
    • git stash pop从堆栈中弹出最上面的储藏库，并将其与你的工作环境合并。
    • 合并一个储藏库肯定会产生冲突，如果有已提交的修改涉及到同一件工作。解决这些冲突的方法与合并两个分支时的方法相同。

    接下来去哪里？¶

    缓存是一种优秀的Git机制，可以帮助你作为一个开发者的生活变得更容易一些。最棒的是，Git同样遵循大多数开发者自然而然的工作流程，比如把尚未完成的改动藏在一边（这比我以前做的那些糟糕的目录拷贝要好得多）。

    接下来的两章是关于Git中一个更有用，但却被广泛误解的操作：重命名。关于合并还是重放是Git的一个更好的策略，现在有很多不同的意见。为了帮助你理解双方的争论，你将深入研究重放，以便你能形成自己的观点，了解什么对你的团队和工作流程最有意义。
